Document Standard — Droppable Prompt   ·   Generated 20260703b   ·   git-versioned; pull for current   (supersedes v0.0.7)

Keep this document to the following standard.

ORDER — Reverse-chronological, newest-first. New entries insert at the TOP of the entry stack, directly below the "How this document is kept" section, pushing older entries down. (Insurance-adjuster file convention.)

ENTRIES — Each entry is stamped with DATE and TIME. Multiple entries per day are expected and fine. Nothing is ever overwritten or removed; the document keeps its full history and only grows. An entry is a PHOTOGRAPH of the flow — one change frozen at one instant.

SUMMARIES — Periodically, write a SUMMARY entry and prepend it like any other entry (it is ADDED, never substituted; older entries stay below it). Each summary is a COMPLETE current-state snapshot — the latest summary alone is the full picture. A summary is a VIDEO of the flow — it shows the flow moving, not frozen.

READING — Scan DOWN to the latest summary; read it for complete current state; then read UPWARD through the entries added since (the not-yet-summarized tail). For deep history, continue downward — all prior summaries and entries remain.

VERSION — major.minor.checkpoint. Major = release-grade milestone (1.0.0 = first public release; leading 0 = pre-public/prototype). Minor = the consolidating move (code: a grouped revision; document: a summary entry). Checkpoint (rightmost) = the atomic landed change (code: a commit, with every file in it carrying that version; document: an entry). New minor zeroes the rightmost; new major zeroes both. This version system governs CODE, where the commit carries the version and files inherit it.

DOCUMENT IDENTIFICATION (supersedes the former version-in-title rule) — Documents are stored in git, which carries their version history; a document does NOT carry a major.minor.checkpoint version in its title. Instead the FILENAME is stable (no version, no date — e.g. "A_Working_Profile_of_Mark.docx"), so git tracks the document as one evolving thing across commits, and the document carries a GENERATION STAMP as the SECOND LINE, directly under the title (footers are not shown by phone .docx readers, so the stamp goes in the body). The stamp is a reverse-date-plus-letter form — YYYYMMDD followed by a single lowercase letter for that day's revision sequence (e.g. "20260703a", "20260703b" for the second generation that day, up to twenty-six a day). Format: "Generated 20260703b · git-versioned; pull for current". Rationale: the letter-date conveys revision sequence at a glance where a raw timestamp does not; git holds the true history underneath; currency is established by pulling, and the stamp is the receipt that the pull was fresh. (The reverse-date-plus-letter scheme is the author's own native document-versioning convention, relocated from filename into the document body so filenames stay stable for git.)

CODE — When changing a code file, deliver the WHOLE file, not a fragment. Present the entire updated file(s) as a complete drop-in replacement ready to upload — never a diff or fragments to hand-splice, because splicing into running code invites transcription error. With the file, provide a ready-to-paste COMMIT DESCRIPTION in this exact shape: the GitHub SUMMARY LINE carries the version number alone (e.g. "v0.2.10"); the EXTENDED DESCRIPTION body leads with a single line of the form "version — summary text" (e.g. "v0.2.10 — Unify to single-column tabbed layout"), followed by the descriptive body. So the version leads both fields; the human-readable summary lives at the top of the extended description, not in the GitHub summary line. This makes the checkpoint document itself and keeps the commit list scannable by version. Version and syntax are part of the delivery: bump the checkpoint (rightmost) per the VERSION rule with every landed change, and confirm the file parses before presenting it. DOCUMENTATION CADENCE: every checkpoint lands as at least one dated entry in Notes, written as the work is done — the commit (or session) is the change; the Notes entry is its photograph in the development record. Notes are kept by plain dated accretion, newest-first (NO review-materialization step — that idea is retired for Notes because it must be run by hand and in practice may not be). Entries go to TWO places by scope-slice: (1) the PROGRAM Notes always gets one entry with the FULL session record — everything, whether or not it belongs to a single project; (2) if the session touched a project, the PROJECT-SCOPED SLICE of that same entry also goes to that project's own Notes (in its repo, beside its code). The project entry is a scope-filtered slice of the program entry, not a separately-authored second document: when the session is purely about one project the slice removes nothing, so the two entries are IDENTICAL text; when the session also ranges wider, the program entry is the project slice PLUS the extra material. One authoritative entry (the full one, to the program); the project version is that entry trimmed to its scope. A session spanning two projects → one program entry and two project slices, done explicitly at checkpoint (no automatic router). The duplication (project work appears in both) is accepted and cheap — it buys co-location, each project's decisions sitting with its code while the program Notes stays the one complete workshop history. (Code files carry their version in the commit and in-app, not appended to a title, and take no postscript.)

REPOS — Two-tier structure. A PROGRAM repo is the workshop, holding all development documents (Purpose, Design, Notes, methodology, standards, profiles). PROJECT repos are destinations, holding a project's code plus the distilled review-documents produced for it. The program develops; projects receive; the flow is one-directional. From the author's side, both project code and project documents are read-only outputs — handed over and placed (in the right repo folder), not hand-edited. Design and functional verification are the author's; authoring the code is the tool's.

POSTSCRIPT — At the end of every document that is a product (not individual source files), add, verbatim: "Prepared with the assistance of and drafting by Claude AI; concepts and direction by J. Mark Weller." Apply uniformly to all such product documents. (Individual source files — code, and functional instruments such as this prompt — are exempt, as they are with the document-generation-stamp rule.)

WHY (do not discard) — What is being designed is a FLOW, not a fixed artifact; the flow is change (motion). An entry photographs a change; a summary videos the flow in motion — which is why the summary is the load-bearing current-state view. The document is never "finished." Keep claims provisional where provisional; mark candidate vs. settled; distinguish observation from inference; add only what must be added — do not pad.
